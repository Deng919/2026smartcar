# GitHub 仓库发布与 README 设计

## 目标

将当前 TC264 智能车 ADS 工程作为可直接导入和继续开发的完整源码工程，发布到空仓库 `Deng919/2026smartcar`。仓库同时提供中文 README，使首次接触项目的人无需逐个阅读源码即可了解硬件平台、模块分工、运行流程、按键操作、参数位置和编译方法。

## 发布方式

当前 GitHub 仓库为空，因此直接在现有工程根目录初始化 Git，并使用 `main` 作为默认分支。保留完整的本地源码目录结构和 ADS 工程描述文件，添加远程地址后进行首次推送，不创建额外的仓库嵌套目录。

提交分为两步：

1. 设计说明提交，用于记录发布范围和约束。
2. 源码发布提交，包含源码、工程配置、README 和 `.gitignore`。

## 纳入仓库的内容

- `code/`：图像识别、电机、PID、舵机和集中参数。
- `user/`：CPU0/CPU1 主程序、中断及用户配置。
- `libraries/`：逐飞开源库、英飞凌 iLLD 和设备驱动，保证工程完整。
- `.project`、`.cproject`、`.settings/`：AURIX Development Studio 工程配置。
- 链接脚本、调试启动配置、引脚说明和工程辅助脚本。
- `docs/`：本次迭代形成的设计与实施记录。
- `README.md`、`.gitignore`。

## 排除内容

- `Debug/`、`Release/` 及其他可重新生成的构建目录。
- `.ads/` 中的本机 winIDEA 工作区状态。
- ELF、HEX、MAP、对象文件、依赖文件和编译中间文件。
- IDE 缓存、操作系统临时文件、Python 缓存和本次同步产生的临时目录。

不排除 `.project`、`.cproject`、`.settings/` 和调试 `.launch` 文件，因为它们是导入 ADS 工程所需的配置，而不是普通 IDE 缓存。

## README 结构

README 使用中文，按以下顺序组织：

1. 项目简介与当前功能。
2. 硬件及开发环境：TC264D、AURIX Development Studio、TASKING、MT9V03X、TFT、编码器、电机和舵机。
3. 目录结构与各应用模块职责。
4. 主运行流程：DMA 完整帧、Otsu、二值化、起点搜索、逐行搜线、丢线保持、中线加权、转向和速度闭环。
5. 控制说明：动态舵机 PD、左右轮差速、弯道降速、速度前馈和增量 PID。
6. 按键说明：KEY_1 启停，KEY_2 开关 TFT 图像显示但保持识别运行。
7. 参数调整入口：集中指向 `code/config.h`，列出主要参数类别，不复制全部数值表。
8. ADS 导入、Debug 构建、下载与首次实车检查步骤。
9. 已知注意事项：赛道白色围栏与黑色间隔可能影响识别，参数需结合摄像头安装和实车测试调整。
10. 成员分工：HPX 负责基础代码构建，DZH 负责代码优化。
11. 开源库与许可证说明，保留原逐飞及 GPL 声明。

## 安全与历史保护

- 推送前先检查 GitHub 登录状态和远程分支状态。
- 远程仓库必须仍为空；如果推送前发现远程已有提交，则停止并重新比较，不强制覆盖。
- 不使用强制推送。
- 提交前检查暂存文件列表和大文件，确认没有 Debug 产物或本机绝对路径泄漏。
- 推送后读取远程 `main` 分支提交，确认远程提交与本地 HEAD 一致。

## 验证标准

- `.gitignore` 能排除所有现有 `Debug/` 文件和 `.ads/` 本机状态。
- Git 暂存列表包含完整源码和 ADS 工程配置，不包含编译产物。
- README 中的路径、按键和参数说明与当前源码一致。
- 源码发布提交前重新执行应用层静态校验：配置参数存在、关键文件存在、六个应用 `.c` 文件可按 GBK 解码。
- `git status` 在提交后干净。
- `git push origin main` 成功，且 `git ls-remote origin refs/heads/main` 返回的提交与本地 HEAD 相同。
